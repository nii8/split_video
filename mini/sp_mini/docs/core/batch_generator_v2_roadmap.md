# Batch Generator V2 三阶段实施路线

## 目标

在现有 `batch_generator.py` 基础上，分三步把系统升级为更先进的自动短视频生成流程：

1. 先加入 **画面检测评分**
2. 再加入 **拼接自然度评分**
3. 最后加入 **多视频组合生成**

这样做的目的不是一次性推翻旧系统，而是分阶段演进。每完成一步，都可以稳定落地、单独验证，然后再继续下一步。

---

## 总体原则

- 每一阶段都要可单独运行、可单独验证
- 每一阶段都尽量复用现有 `make_time/`、`make_video/`、`batch/` 模块
- 每一阶段都把“高成本判断”尽量前移，避免无意义生成
- 每一阶段结束后，都要输出结构化缓存和总结文件，方便下一阶段继续读取

---

# 第一阶段：画面检测评分

## 这一阶段要解决什么问题

当前系统主要看字幕、脚本、时间轴，但没有判断：

- 这段画面是否清晰
- 人脸是否好看、可出镜
- 光线是否合适
- 表情是否有感染力
- 肢体动作是否自然
- 是否有明显空镜、低头、模糊、背景杂乱

结果就是：

- 有些文本很好，但画面很差，最终成片不好看
- 有些候选其实在画面层面就该提前淘汰，却拖到最终视频阶段才发现

这一阶段的目标，就是把“画面质量判断”前移到视频生成之前。

---

## 核心思路

当某个候选脚本已经完成时间轴匹配，拿到了 `intervals` 之后：

1. 对每个候选片段按固定时间间隔抽帧
2. 把一段时间内的多张图拼成 9 宫格
3. 调用多模态 LLM 对这 9 张图做一次整体评分
4. 得到该片段的视觉质量分
5. 汇总整条候选视频的视觉总分

推荐默认策略：

- 每 `2` 秒取一帧
- 一个 `18` 秒窗口对应 `9` 张图
- `9` 张图合成为一张 9 宫格图
- 多模态 LLM 对该窗口做一次请求

如果后续觉得太粗，可以升级到：

- 每 `1` 秒一帧
- 10 分钟视频约 60 次请求

但第一阶段建议先用 `2` 秒方案，控制成本。

---

## 这一阶段插入到流程的哪里

建议放在：

1. 文本筛选
2. 脚本生成
3. 时间轴匹配
4. **画面检测评分**
5. 最终候选排序
6. 生成视频

也就是说，画面检测应发生在 `intervals` 已经确定之后、正式生成视频之前。

原因：

- 只有先知道用了哪些时间段，才能知道具体抽哪些帧
- 这时做画面评分最准确，也最省

---

## 第一阶段要做的模块

建议新增：

- `batch/visual_scorer.py`
- `batch/frame_sampler.py`
- `batch/image_grid.py`

建议新增缓存目录：

```text
data/batch_results/{video_id}/visual/
├── frames/
├── grids/
├── visual_scores.json
└── selected_windows.json
```

---

## 第一阶段要输出什么

每个候选视频要新增一组视觉评分结果，示例：

```json
{
  "candidate_id": "i_003",
  "visual_score": 8.4,
  "windows": [
    {
      "window_index": 1,
      "start_sec": 0,
      "end_sec": 18,
      "score": 8.8,
      "issues": ["light_ok", "expression_good"],
      "summary": "人脸清晰，表情有感染力，构图稳定"
    }
  ]
}
```

---

## 第一阶段验收标准

- 系统能对 `intervals` 对应的视频窗口自动抽帧
- 系统能生成 9 宫格图
- 系统能调多模态 LLM 返回视觉评分
- 候选视频总分里新增 `visual_score`
- 明显画面差的候选会被降分或淘汰

---

## 第一阶段完成后，进入第二阶段前要复查什么

完成第一阶段后，先确认下面几件事：

1. 视觉评分是否真的能筛掉差画面片段
2. 请求量是否可接受
3. 9 宫格抽样密度是否合适
4. 是否需要加入更多视觉维度

确认通过后，再进入第二阶段。

---

# 第二阶段：拼接自然度评分

## 这一阶段要解决什么问题

即使文本好、单段画面也好，拼起来仍可能不自然，比如：

- 画面突然跳动很厉害
- 人物位置变化太大
- 镜头角度断裂
- 动作衔接不连贯
- 场景变化太猛
- 连续多个短片段切换，造成“抽搐感”

这种问题如果不单独评分，会导致最终视频有明显拼接痕迹。

这一阶段的目标，就是判断：

“这些片段单独都不错，但拼在一起是否顺滑自然？”

---

## 核心思路

拼接自然度分两层：

### 第一层：规则评分

先做低成本快速判断：

- 总转场次数是否过多
- 平均片段长度是否过短
- 是否有连续多个极短片段
- 相邻片段时间跨度是否过大
- 视频整体节奏是否过碎

### 第二层：多模态拼接评分

再对关键切点做视觉判断：

1. 取切点前后若干帧
2. 拼成小图组
3. 调用多模态 LLM 判断切换是否自然
4. 给每个切点打一个“跳切惩罚分”

最后整条候选视频得到：

- `transition_score`
- `jump_cut_penalty`
- `stitch_naturalness_score`

---

## 这一阶段插入到流程的哪里

建议放在视觉评分之后：

1. 文本筛选
2. 脚本生成
3. 时间轴匹配
4. 画面检测评分
5. **拼接自然度评分**
6. 候选汇总排序
7. 生成视频

原因：

- 只有先知道每段具体视频片段，才能判断切点前后画面变化
- 拼接自然度本质上依赖真实切点，不适合前置

---

## 第二阶段要做的模块

建议新增：

- `batch/transition_scorer.py`
- `batch/cutpoint_sampler.py`

建议新增缓存目录：

```text
data/batch_results/{video_id}/transition/
├── cutpoints/
├── transition_scores.json
└── jump_cut_reports.json
```

---

## 第二阶段要输出什么

示例：

```json
{
  "candidate_id": "i_003",
  "transition_score": 7.6,
  "jump_cut_penalty": 1.5,
  "cutpoints": [
    {
      "cut_index": 2,
      "score": 5.2,
      "issues": ["camera_shift_large", "subject_position_jump"],
      "summary": "前后镜头切换突兀，人物位置变化明显"
    }
  ]
}
```

---

## 第二阶段验收标准

- 系统能识别每个候选视频的切点
- 系统能对切点前后抽帧并做评估
- 系统能对“跳切严重”的候选自动扣分
- 候选总分里新增 `transition_score`
- 最终排序明显偏向拼接更自然的方案

---

## 第二阶段完成后，进入第三阶段前要复查什么

完成第二阶段后，先确认：

1. 跳切严重的候选是否真的被压下去了
2. 规则评分和多模态评分是否一致
3. 是否需要减少多模态切点评估次数
4. 拼接自然度在总分中的权重是否合理

确认通过后，再进入第三阶段。

---

# 第三阶段：多视频组合生成

## 这一阶段要解决什么问题

当前系统默认一条短视频只来自一个原视频。

但真实更高级的需求是：

- 一条短视频可以由多个视频组成
- 第一个视频取一段
- 第三个视频取两段
- 第二个视频取结尾

也就是说，不再是“单视频剪辑”，而是“跨视频编排”。

这一阶段的目标，是让系统支持：

- 两个视频甚至多个视频共同生成一条主题统一的短视频

---

## 核心思路

多视频模式分两层：

### 第一层：每个视频先独立建素材池

对每个视频分别做：

- 文本分析
- 时间轴匹配候选
- 画面评分
- 拼接评分基础信息

每个视频先产出自己的候选片段池。

### 第二层：跨视频重组

系统再从多个素材池中抽片段，拼成一条新视频。

例如主题是“失败后重新站起来”：

- 视频 A：失败经历
- 视频 C：认知转折
- 视频 B：结尾升华

系统根据主题、文本、画面和兼容性来组织这些片段。

---

## 多视频模式新增的关键判断

多视频组合不能只看文本，还必须多看几种兼容性：

- 主题一致性
- 情绪一致性
- 画面风格一致性
- 主体一致性
- 节奏一致性
- 切换是否突兀

否则虽然内容相关，但拼起来会像“杂剪合集”。

---

## 这一阶段插入到流程的哪里

这不是在原流程中插一个评分器，而是把“候选生成”本身扩展成两种模式：

- 单视频模式
- 多视频模式

新的流程可以理解为：

1. 先对多个视频分别做素材分析
2. 为每个视频建立候选片段池
3. 如果是单视频任务，就在单池中组装
4. 如果是多视频任务，就跨池组装
5. 然后再进入视觉评分、拼接评分、最终排序

---

## 第三阶段要做的模块

建议新增：

- `batch/multi_video_planner.py`
- `batch/multi_video_selector.py`
- `batch/cross_video_scorer.py`

建议新增缓存目录：

```text
data/batch_results/multi_video/
└── {task_id}/
    ├── source_videos.json
    ├── pooled_segments.json
    ├── composition_candidates.json
    ├── cross_video_scores.json
    └── summary.json
```

---

## 第三阶段要输出什么

示例：

```json
{
  "task_id": "mv_001",
  "theme": "失败后重新出发",
  "candidate_id": "mv_c_003",
  "segments": [
    {"video_id": "A001", "start": "00:00:21,000", "end": "00:00:30,000"},
    {"video_id": "C003", "start": "00:01:12,000", "end": "00:01:25,000"},
    {"video_id": "B002", "start": "00:03:10,000", "end": "00:03:20,000"}
  ],
  "cross_video_score": 8.1,
  "issues": ["style_gap_small"],
  "summary": "主题一致，跨视频切换较自然，适合作为多源组合候选"
}
```

---

## 第三阶段验收标准

- 系统支持同时读取多个视频及对应 SRT
- 系统能先为每个视频建立候选片段池
- 系统能跨视频选片段重组一条候选视频
- 系统能给多视频候选打兼容性分
- 最终生成的视频在主题上统一、画面上基本可接受

---

## 第三阶段完成后系统会变成什么

做到这一步之后，系统就不只是“自动剪辑器”了，而更像一个“自动短视频编排引擎”。

它会同时做三件事：

- 选内容
- 选画面
- 选转场和素材来源

这比当前只依赖字幕和时间轴的版本更高级。

---

# 当前阶段补充说明（2026-04-13）

V2 三阶段从“开发打通”的角度看，主体已经基本完成。

但当前项目并没有停在“功能已接入”这里，而是进入了新的主线阶段：

## 当前真正目标

当前真正目标已经变成：

1. 按时长分布批量生成样本
2. 单视频每源稳定生成固定数量
3. 多视频稳定生成大批量候选
4. 输出机器评分表
5. 输出人工评分模板
6. 用人工回标继续调整评分算法

也就是说，现在的核心不再只是“第三阶段有没有接上”，而是：

**这套系统能不能形成“批量出样本 -> 机器评分 -> 人工评分 -> 调算法”的闭环。**

## 当前阶段的现实边界

当前仍然不做的事：

- 不直接追求通用型视频平台
- 不急着做复杂导演逻辑
- 不急着做全自动爆款预测
- 不急着引入复杂训练管线

当前优先做的是：

- 先让每一轮测试都稳定产出可用样本
- 先让机器评分和人工评分能稳定对齐
- 先让算法迭代建立在真实回标数据上

## 对未来 V3 的更现实定义

如果进入 V3，更合理的方向不是盲目扩模块，而是：

1. 评分维度学习
   - 根据人工回标修正评分维度和权重
2. 风格模板化
   - 针对赛道沉淀“偏好长度 / 节奏 / 编排模式”
3. 样本库积累
   - 保存高人工分样本，形成可复用参考集
4. 候选生成策略升级
   - 不再只按规则拼接，而是开始学习什么样的组合更容易高分

---

# 推荐实施顺序

严格按下面顺序推进：

1. 第一阶段：画面检测评分
2. 第二阶段：拼接自然度评分
3. 第三阶段：多视频组合生成

原因：

- 视觉评分是单视频模式下最直接、收益最大的增强
- 拼接自然度建立在已有视频片段和视觉信息之上
- 多视频组合复杂度最高，必须放最后

---

# 每个阶段结束后要做什么

## 第一阶段结束后

- 回读本文件“第二阶段”部分
- 检查视觉评分输出是否稳定
- 再进入拼接自然度评分开发

## 第二阶段结束后

- 回读本文件“第三阶段”部分
- 检查跳切惩罚是否有效
- 再进入多视频组合开发

## 第三阶段结束后

- 回读本文件总体目标
- 再决定是否进入 V3，例如：
  - 自动学习成功样本
  - 风格模板库
  - 基于历史爆款结果自动调权重

---

# 一句话总结

这三步分别是把系统升级成：

1. **会看画面**
2. **会看转场**
3. **会跨视频编排**

按这个顺序推进，复杂度最低，也最容易每一步都看到明确收益。
