# V2 第三阶段验收状态报告

## 日期
2026-04-13

## 定位

这不是“项目彻底完成报告”，而是当前批量生成系统 V2 第三阶段的最新验收状态说明。

当前更准确的结论是：

- 第三阶段主功能已经打通
- 多视频候选到实际视频生成的链路已经接入
- 但最新综合测试已经证明，当前还不能把“有多视频 mp4 输出”视为“验收通过”

## 最新测试结论

根据最新综合测试报告与生成视频统计：

- 单视频共生成 225 个视频
- 多视频共生成 3 个视频
- 多视频文件时长分别为：
  - `4.834s`
  - `7.000s`
  - `2.034s`
- 多视频平均时长仅 `4.62s`

这说明：

1. 多视频输出链路是通的
2. 但当前候选和评分逻辑仍可能偏好过短片段
3. 原测试报告口径把“能生成 mp4”误当成了“测试通过”

## 当前验收结论

### 已完成部分

1. 第一阶段视觉评分已接入主流程
2. 第二阶段拼接自然度评分已接入主流程
3. 第三阶段多视频组合流程已接入实际输出逻辑
4. `batch_generator.py` 已能为高分多视频候选调用 `generate_multi_video()`
5. 多视频输出目录统一落在 `data/batch_results/multi_video/`
6. 生成结果已按 `run_id` 隔离到独立目录，降低多次运行互相覆盖的风险
7. 已补最小硬保护：
   - 无效片段直接跳过
   - 源视频缺失时直接报错
   - 无有效输入路径时直接报错

### 当前未通过项

1. **成片时长未达标**
   - 当前多视频输出未满足“必须大于 18 秒”的交付门槛
2. **测试报告口径过宽**
   - 不能再把“视频文件有效”直接判定为“功能通过”
3. **候选偏好仍需收紧**
   - 候选生成与评分需要明确偏向时长达标方案

## 当前阶段判断

当前阶段应定义为：

**第三阶段开发基本完成，但验收尚未通过；当前核心阻塞是多视频输出时长不达标。**

而不是：

**整个项目已经彻底完成，无需继续收口。**

## 本轮收口重点

1. 把 `18s` 最小时长门槛前移到候选生成和评分逻辑
2. 生成阶段继续保留最小时长硬过滤
3. 测试脚本与测试报告必须把 `<18s` 直接视为失败
4. 保持默认配置安全，不让多视频模式自动影响原单视频流程
5. 保持运行产物不进入 git
6. 保持文档结论和实际代码状态同步

## 一句话结论

V2 第三阶段已经做成了“能生成”，但还没有通过“能交付”的验收；当前最核心的验收门槛是：

**多视频输出必须大于 18 秒，且测试报告必须真实反映这一点。**
