## 二、用户当前明确偏好

这些偏好会直接影响后续实现方式。

### 1. 目标优先，代码哲学从简

当前不是追求“漂亮代码”“完美抽象”“未来感架构”，而是：

- 在有限时间内把目标功能做出来
- 不浪费 token
- 不浪费电力
- 每一段代码都尽量有明确用途

### 2. 不要花里胡哨

倾向：

- 过程式
- 直白
- 少抽象
- 少多余层
- 少无意义包装

不倾向：

- 为了优雅而优雅
- 为了以后也许扩展而提前搭很重的结构
- 一上来就做完整框架

### 3. 不要堆无意义的防御式代码

用户明确强调：

- 如果某个 `try/except` 在真实生产运行里 99.999% 不会触发，而且失败后人工重启就能解决，那没必要写
- 不要让大量代码变成“几乎永远运行不到的防御层”

所以后续实现倾向：

- 保留必要的关键边界检查
- 不写大量低价值 try/except 包装
- 错了就让它暴露，方便快速修正

### 4. 允许小错，不允许大错

当前容忍度是：

- 允许存在小比例实现细节错误
- 后续测试阶段由 Opencode 发现并修复

但不允许：

- 战略方向错
- 架构根基错
- 数据流设计错
- 主链路闭环设计错
- 产生后续难以收拾的结构性事故

也就是说：

**当前最重要的是避免“大的、本质性的错误”。**

## 四、项目当前目标

项目的核心目标可以概括为：

**把原来的“单次自动剪辑工具”逐步升级成一个自动短视频编排引擎。**

它不是只做：

- 给一个视频剪一个成片

而是逐步做成：

- 从已有视频池里自动生成大量候选
- 自动评分

- 逐步加入视觉判断、结构判断、跨视频编排能力

## 七、当前代码结构理解

### 1. 单视频主链路

当前单视频主链路大体是：

1. 输入原视频和 SRT
2. Phase1：LLM 从字幕中筛选高价值内容
3. Phase2：LLM 生成短视频脚本
4. Phase3：脚本匹配到字幕时间轴，得到 `keep_intervals`
5. Phase4：评分
6. Phase5：调用 `cut_video_main()` 输出视频

### 2. 当前多视频方向的理想链路

我们希望的第三阶段多视频链路是：

1. 多个视频分别跑 phase1 -> phase2 -> phase3
2. 每个视频形成自己的候选片段池
3. 从多个池中挑片段组成候选
4. 对候选做跨视频基础兼容评分
5. 再决定是否输出成最终视频

### 3. 当前真实情况

现在第三阶段代码已经有了：

- `batch/multi_video_selector.py`
- `batch/video_pool_builder.py`
- `batch/video_combiner.py`
- `batch/multi_video_scorer.py`
- `batch_generator.py` 中的多视频入口

但当前真实问题是：

- 还没有和旧系统的真实返回结构完全对齐
- 还没有形成真正可运行的最小闭环
- 当前更接近“已搭骨架”，不等于“已联通”

## 八、当前最关键的架构问题

Claude 需要重点评估这些问题是不是结构性风险。

### 问题 1：多视频层是否应该建立在旧 `keep_intervals` 之上

现在旧系统里最稳定的输出结构是：

- `keep_intervals`

它已经是：

- 单视频 phase3 的真实结果
- 单视频输出函数 `cut_video_main()` 的真实输入

当前风险点是：

- 第三阶段多视频模块想使用更适合组合的 `segment dict`
- 但新结构和旧结构还没有稳定转换层

Claude 需要评估：

- 最合理的做法，是不是应该把 `keep_intervals` 作为底层稳定格式保留
- 然后在多视频层上增加一个非常薄的转换层
- 而不是试图让第三阶段直接绕开旧格式

### 问题 2：最终输出层是否应该立即实现真正多视频拼接

当前 `cut_video_main()` 本质上还是单视频输出函数。

如果第三阶段要真正产出跨多个 mp4 的成片，理论上需要：

- 一个新的多输入视频拼接出口
- 或一个清楚的最小降级策略

Claude 需要评估：

- 当前阶段是否值得立即实现真正多视频输出函数
- 还是应该先把第三阶段收口成：
  - 能生成并排序多视频候选
  - 先输出候选 summary
  - 真正多视频输出放下一步

### 问题 3：第三阶段当前最小闭环应该闭到哪里

这是现在最重要的问题。

有两种可能的“最小闭环”：

#### 方案 A：闭到候选结果

做到：

- 每个视频形成片段池
- 多视频候选生成出来
- 有评分
- 有 summary
- 但暂不做真正跨视频视频输出

优点：

- 简单
- 风险低
- 能快速验证架构方向

缺点：

- 还不是最终完整功能

#### 方案 B：闭到最终视频输出

做到：

- 连最终多视频成片都产出

优点：

- 功能完整

缺点：

- 风险更高
- 复杂度更高
- 更可能把当前阶段拖慢

Claude 需要重点评估：

**当前阶段最合理的最小闭环，应该停在 A，还是应该直接做 B。**

## 九、我当前的实现倾向

基于当前项目状态、用户偏好和时间预算，我当前更倾向下面的实现策略。

### 1. 继续保持过程式、扁平化实现

后续代码倾向继续保持：

- 少 class
- 少抽象
- 少新层
- 主流程直白
- 小函数配合主流程展开

### 2. 先追求“可联通”，再追求“完整优雅”

当前优先级：

- 先让多视频主链路可联通
- 再看是否有必要上真正多视频输出
- 不先做复杂导演系统

### 3. 新结构建立在旧稳定接口上，而不是彻底推翻旧结构

我倾向：

- 保留旧的 `keep_intervals` 作为稳定底层产物
- 新增一个明确、轻量的 `keep_intervals -> segment` 转换层
- 多视频层只在转换后的结构上工作

### 4. 尽量减少低价值防御性代码

除非边界非常关键，否则：

- 不堆 try/except
- 不做一堆“几乎不会触发”的防守代码
- 出错就让它尽快暴露，便于修正

### 5. 测试与实现分工明确

- 我负责主实现
- Opencode 负责测试、跑链路、反馈问题、修小错

这样能让主链路收敛得更快。
