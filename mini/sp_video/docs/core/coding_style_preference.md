# Coding Style Preference

## 目的

这份文档只保留当前最重要的代码偏好。

目标不是写得花，而是：

- 目标优先
- 主链路优先
- 少浪费 token
- 少浪费时间
- 代码尽量有明确用途

---

## 总体原则

- 过程式优先
- 主流程直接展开
- 少抽象
- 少 class
- 少无意义中间层
- 先把事情做成，再谈精致

---

## 当前最重要的偏好

### 1. 先做有用的代码

优先写：

- 真正参与主链路的代码
- 真正会运行到的代码
- 真正能缩短实现路径的代码

少写：

- 只是为了好看存在的代码
- 只是为了“以后也许扩展”提前铺的代码
- 很久都用不到的抽象层

### 2. 主流程要一眼能看懂

倾向：

- 一个文件做一段清楚的事
- 变量名直白
- 主逻辑按步骤展开
- 小函数为主流程服务

不倾向：

- 看半天还不知道主流程在哪里
- 为了整齐拆出很多空层
- 把简单逻辑包得很绕

### 3. 少 class，少框架感

优先用：

- 函数
- `if`
- `for`
- `dict`
- `list`
- 少量必要的辅助函数

除非真的有明显收益，否则不要引入 class。

### 4. 命名要直白

喜欢这种名字：

- `get_keep_intervals`
- `cut_video_main`
- `build_multi_video_pools`
- `score_multi_video_candidate`

少用太抽象的名字：

- `resolve_context`
- `build_pipeline`
- `process_data`

### 5. 允许适度重复，换可读性

如果抽公共层会让代码更绕，那可以保留少量重复。

优先级：

1. 好读
2. 好改
3. 好查错
4. 再考虑复用

### 6. 少写低价值 try/except

不要默认到处包 `try/except`。

倾向：

- 只在关键边界写必要保护
- 对真正常见的问题做处理
- 对明显不会发生的情况，不额外堆防御代码

如果某种异常在真实运行里几乎不会发生，而且失败后人工重启就能解决，通常没必要专门为它写一层低价值保护。

### 7. 注释解释“这里在干什么”

注释要帮人快速理解主逻辑。

适合写：

- 这里是在把旧结构转成多视频片段结构
- 这里是在给候选补视觉评分
- 这里是为了把太碎的候选压下去

不需要：

- 每行都解释
- 废话式注释

---

## 一句话总结

代码要直、少、清楚、够用，优先把主链路做通，不要把资源花在低价值的花活和过度防御上。
