# Coding Style Preference

## 目的

这份文档不是通用规范，是当前仓库代码风格偏好的简版说明。

后面继续写代码时，先按这份文档对齐风格。

---

## 总体风格

- 过程式优先
- 脚本式优先
- 主流程直接展开
- 少抽象
- 少 class
- 先让人能看懂

---

## 明确偏好

### 1. 少 class

优先用这些：

- 函数
- `if`
- `for`
- `try/except`
- `print`
- `dict/list`

除非真的必要，不要引入 class。

---

### 2. 一个文件做一件清楚的事

不要为了“分层整齐”拆出很多空中间层。

更偏向：

- 一个文件负责一段完整逻辑
- 主流程一眼能看懂
- 看函数名就知道在干什么

---

### 3. 命名要直白

喜欢这种名字：

- `get_keep_intervals`
- `cut_video_main`
- `parse_zimu_content`
- `score_interval_candidate`

少用太抽象的名字，比如：

- `process_data`
- `build_context`
- `resolve_pipeline`

---

### 4. 允许适度重复，换可读性

如果拆成公共函数反而更绕，那可以不拆。

优先级是：

1. 顺着看得懂
2. 调试方便
3. 再考虑复用

---

### 5. 中间变量要看得懂

宁可多两三行，也不要把逻辑压成一行。

常见可接受变量名：

- `start_sec`
- `end_sec`
- `video_path`
- `srt_path`
- `keep_intervals`
- `candidate_list`

---

### 6. 错误处理偏保守

常见写法：

- `try/except`
- `print(...)`
- 失败返回 `None` / `False` / 空列表

不要先设计复杂异常体系。

---

### 7. 注释解释“在干什么”

关键步骤前最好有一句中文说明。

比如：

- 这里是在抽帧
- 这里是在匹配字幕
- 这里是为了防止结果太碎

不用每行都注释，但关键块前最好有人话。

---

### 8. 留白可以有，但别过度设计

可以接受：

- 少量配置
- 少量全局变量
- 明确标注“后续运行环境再补”

不建议：

- 为了以后可能扩展，先搞很重的配置系统
- 一个简单功能先抽成复杂框架

---

## 写新代码时的建议

1. 先写主流程
2. 再拆少量小函数
3. 少用 class
4. 数据结构优先 `dict/list`
5. 关键地方加一两句中文注释
6. 先可读，再考虑整洁

---

## 一句话总结

代码要直、清楚、过程式、容易 review、容易 debug。
