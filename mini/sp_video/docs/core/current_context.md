## 当前项目定位

`mini/sp_video` 已经不再只是：

- 输入长视频 + SRT
- 输出 1 条短视频

它当前更接近：

**自动短视频候选生成、评分、筛选、逐步走向多视频编排的本地批处理系统。**

当前真正主线是：

- `batch_generator.py`
- 批量生成候选
- 自动评分
- 筛选高分结果
- 继续推进 V2 第三阶段：多视频组合

`skill.py` 和旧 CLI 仍然有价值，但已经不是当前主战场。

## 当前最重要的判断

### 1. 当前目标已经明确

当前不再继续发散设计。

当前最重要的事已经从“打通第三阶段”转成：

**对第三阶段做验收加固，让它从“已经做通”变成“默认安全、可复现、可交付”。**

### 2. 当前主要阻塞已经从“闭环无法验证”变成“是否继续进入下一阶段”

之前的真实阻塞点是：

- Phase1 / Phase2 的 AI 调用太慢
- 默认批量参数太重

已知根因：

- 模型单次响应约 `76-91` 秒
- 默认参数：
  - `BATCH_PHASE1_COUNT = 20`
  - `BATCH_PHASE2_COUNT = 100`
- 完整链路接近数小时

为了解决这个问题，当前已经引入：

- `BATCH_TEST_MODE`
- 或等价的降载测试方式

根据最新测试报告：

- 多视频最小闭环已经在测试模式下完整跑通
- 3 个视频成功跑完 `Phase1 -> Phase2 -> Phase3`
- 成功生成 20 个多视频候选
- `summary.json` 输出正确
- 测试后参数可恢复

所以当前主问题不再是“最小闭环能不能验证”，也不再是“要不要继续做多视频输出层”，而是：

- 最近多视频实现是否足够可靠
- 默认配置是否安全
- 运行产物和测试资产是否放在正确位置
- 完成报告是否过度乐观

### 3. 当前最小闭环边界已经确定

当前第三阶段先采用：

**方案 A：闭到候选 + 评分 + summary，不闭到最终多视频视频输出。**

原因：

- 风险更低
- 更容易验证方向是否正确
- 能避免过早卷入多输入 ffmpeg 输出层复杂度

## 固定提示词设计原则

固定提示词的目标不是短，而是稳定、可复用、低歧义。

以后固定提示词必须明确：

- 唯一项目根目录
- 文档读取优先级
- “按修改时间找最新相关任务”的规则
- 输出格式
- 允许做什么
- 不允许做什么

否则容易出现：

- 读错旧任务
- 读错目录
- 写错 report 位置
- 基于历史状态误推进
