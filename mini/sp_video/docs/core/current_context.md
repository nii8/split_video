## 当前项目定位

`mini/sp_video` 已经不再只是：

- 输入长视频 + SRT
- 输出 1 条短视频

它当前更接近：

**自动短视频候选生成、评分、筛选，并已接通多视频编排输出的本地批处理系统。**

当前真正主线是：

- `batch_generator.py`
- 按时长分布批量生成候选
- 输出机器评分
- 输出人工回标模板
- 用人工评分反推算法
- 对已接通的 V2 第三阶段多视频输出做验收收口

`skill.py` 和旧 CLI 仍然有价值，但已经不是当前主战场。

## 项目演进脉络

主线可概括为：

`单视频 CLI -> skill.py 智能体接口 -> batch_generator 批量生成系统 -> V2 三阶段增强 -> 当前进入第三阶段验收收口`

其中：

- 第一阶段：视觉评分
- 第二阶段：拼接自然度评分
- 第三阶段：多视频组合生成

第一、第二阶段已经接入主流程。
第三阶段开发基本完成，当前不再以“继续扩功能”为主，而是以“批量出样本、回标、迭代评分算法”为主。

## 当前最重要的判断

### 1. 当前目标已经明确

当前不再继续发散设计。

当前最重要的事已经从“打通第三阶段”转成：

**对第三阶段做验收加固，并把系统推进到“可批量出样本、可人工回标、可持续迭代算法”的阶段。**

### 2. 当前主要阻塞已经从“闭环无法验证”转成“验收口径与交付门槛是否真实一致”

之前的真实阻塞点是：

- Phase1 / Phase2 的 AI 调用太慢
- 默认批量参数太重

为解决这个问题，当前已经引入：

- `BATCH_TEST_MODE`
- 或等价的降载测试方式

根据 2026-04-12 晚最新综合测试与视频统计报告：

- 多视频测试脚本表面报告“通过”
- 单视频共生成 225 个视频
- 多视频共生成 3 个视频
- 但这 3 个多视频成片时长分别只有 `4.834s`、`7.000s`、`2.034s`
- 多视频平均时长仅 `4.62s`

所以当前主问题已经进一步收敛为：

- 多视频候选和评分是否在偏好过短片段
- 18 秒最小时长门槛是否真正贯穿候选、生成、测试报告三层
- 测试脚本是否把“有 mp4 文件”误判成“验收通过”
- 单视频和多视频是否能按目标数量与时长分布稳定出样本
- 机器评分字段是否足够稳定，能直接接住人工回标
- 默认配置是否安全
- 运行产物和测试资产是否放在正确位置

### 3. 最新测试目标已切换到“批量出样本 + 人工回标”

本轮不再只追求“跑通一条”。

当前测试目标已经明确为：

- 每个单视频源按时长分布生成 `10` 条视频
- 对列表里的所有单视频源都执行一遍
- 多视频组合目标生成 `100` 条候选成片
- 测试报告必须输出时间统计表
- 测试报告必须输出机器评分表
- 后续由人工对全部视频做多维评分，再与机器评分对比，继续改进算法

### 4. 当前中期目标和终极方向也已经更清楚

当前中期目标不是简单“多生成一些视频”，而是：

- 建立稳定的批量出样本系统
- 建立稳定的机器评分维度
- 建立稳定的人工回标机制
- 形成“生成 -> 机器评分 -> 人工评分 -> 调算法”的闭环

当前更长远的终极方向是：

**把 `sp_video` 从一套批量脚本，推进成面向特定赛道的自动短视频样本工厂与编排引擎。**

### 5. 当前对齐后的目标表述

为了避免后续表述继续发散，当前统一使用下面这组中性表述：

- 终极目标：
  - 做成一个面向目标赛道的自动出片系统
  - 它不只负责生成视频，还负责承接机器评分、人工回标和后续算法迭代
- 近期目标：
  - 先把“批量出片 + 机器评分 + 人工回标 + 改算法”这条闭环跑顺
  - 先让单视频和多视频都能稳定按目标数量和时长分布产出样本
- 当前重点：
  - 不继续发散扩功能
  - 先把样本生成、评分、回标、迭代这条主线收口到稳定可复用

## 当前最该做的事

按优先级排序：

1. 把“多视频输出必须大于 18 秒”前移为硬约束，而不是事后观察指标
2. 收紧测试报告口径，禁止把 `<18s` 的多视频结果判定为通过
3. 严格审核最近多视频实现的代码和文档
4. 收紧默认行为，避免多视频模式自动影响原单视频流程
5. 规范运行产物、忽略规则、示例脚本和验收文档
6. 继续只修高风险点，不做发散重构
7. 让测试报告直接服务后续人工评分回流，而不是只给“通过/失败”结论
8. 在完成一轮稳定回标后，再根据人工分和机器分偏差决定下一轮算法调整

当前不优先做的事：

- 不继续扩新功能
- 不继续扩新文档体系
- 不继续重写评分哲学
- 不做复杂导演逻辑
- 不做低价值重构
