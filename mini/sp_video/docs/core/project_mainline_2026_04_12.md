# 项目主线与行为轨迹

更新时间：2026-04-12 07:30（Asia/Shanghai）

## 当前主线

这个项目的主线已经很清楚了：

- 最早是“长视频 + SRT -> 自动剪成短视频”的 CLI 工具
- 随后扩成给 OpenClaw / 机器人调用的 `skill.py` 流程
- 在 2026-03-29 之后，明显转向“批量预生成 + 自动评分 + 候选筛选”的系统
- 到 2026-04-12 早上，当前真正的工作中心已经不是 `skill.py`，而是 `batch_generator.py` 这条自动批量生成链路，并且正在往 V2 方向推进

## 按时间还原行为轨迹

### 2026-03-22

`TEST.md` 记录了 `skill.py` 的端到端测试结果。

这个阶段重点是把下面这套技能接口跑顺：

- 列视频
- 下载素材
- 生成脚本
- 匹配时间轴
- 生成视频并上传

说明当时项目的核心是给外部智能体调用。

### 2026-03-26

开始补测试计划和单元测试计划。

说明项目开始从“能跑”转向“需要稳住基础逻辑”。

### 2026-03-28

视频剪辑底层从旧方案转向 `ffmpeg filter_complex` 单次调用方案。

这是一次重要的执行链路简化，为后面的批量生成打基础。

### 2026-03-29

项目重心转向批量预生成系统。

核心思路变成：

- 大量生成候选
- 自动打分
- 只保留高质量结果

也就是从“单次交互生成”转向“自动候选工厂”。

### 2026-04-11 晚上

明确了 `batch_generator` V2 三阶段路线：

1. 视觉评分
2. 拼接自然度评分
3. 多视频组合生成

说明后续迭代已经不是零散加功能，而是在按明确路线推进。

### 2026-04-11 深夜到 2026-04-12 凌晨

第一阶段和第二阶段的最小骨架文档、交付说明、交接文档、任务文档陆续补齐。

说明当前阶段的工作方式是：

- 先把骨架写清楚
- 再在运行环境补联通和修问题

### 2026-04-12 早上

第三阶段“多视频组合最小骨架”被提交到代码库。

说明项目已经进入 V2 的第三阶段。

## 当前真实意图

当前最真实的意图不是继续打磨旧 CLI，也不是优先扩 `skill.py`，而是把 `batch_generator.py` 逐步升级成一个：

- 会批量生成候选
- 会自动评分排序
- 会加入视觉判断
- 会加入拼接自然度判断
- 开始具备跨视频选材能力

也就是往“自动短视频编排引擎”的方向走。

## 当前最应该做的事

如果继续推进，优先级应当是：

1. 验证并修通第一阶段、第二阶段在真实运行环境里的链路
2. 把第三阶段多视频骨架修成真正可联通的最小链路
3. 持续更新交接文档，确保后续接手成本低

## 一句话结论

当前项目主线是：

`CLI 原型 -> skill.py 智能体接口 -> batch_generator 批量预生成系统 -> V2 三阶段增强 -> 当前进入多视频骨架阶段`
