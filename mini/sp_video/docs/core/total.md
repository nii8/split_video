## 最高优先级规则：代码写完后必须先 Push，再允许测试

这是当前协作里的最高优先级执行规则之一。

规则内容：

- 只要代码已经写完，并且下一步要交给别人测试
- 就必须先确认代码已经 push 到远端
- 没有 push，不允许进入“让别人测试”的阶段

原因非常直接：

- 如果代码没 push，上游机器拉不到最新实现
- 那么后续测试就是白测
- 会浪费测试机器时间
- 会浪费 token
- 会浪费用户时间
- 还会让用户回来花额外时间排查“为什么测不到最新代码”

这类问题不是小问题，而是：

- 协作事故
- 资源浪费
- 高优先级流程错误

所以以后必须执行下面这个顺序：

1. 写代码
2. 本地自查
3. 提交 git
4. push 到远端
5. 明确确认远端已有对应提交
6. 再通知别人开始测试

如果第 4 步没有完成，就不能让：

- Opencode 测试
- Claude 评估最新实现
- 任何其他机器继续后续流程

一句话：

**没有 push，就等于没有交付；没有交付，就不允许进入测试阶段。**

## 一、项目当前定位

当前项目 `mini/sp_video` 的本质，不再只是一个“输入视频 + 字幕 -> 输出一条短视频”的 CLI 工具。

它已经演进为：

**一个自动短视频候选生成、评分、筛选、输出的本地批处理系统。**

当前真正的主线是：

- 批量生成候选
- 自动评分
- 筛选高质量结果
- 逐步增强视觉判断、拼接判断、多视频组合能力

也就是说，它正在从：

- 单视频自动剪辑工具

演进为：

- 自动短视频编排与筛选引擎

---

## 二、项目主线

项目主线可以概括为：

`CLI 原型 -> skill.py 智能体接口 -> batch_generator 批量预生成系统 -> V2 三阶段增强 -> 当前进入多视频骨架阶段`

### 关键阶段

#### 1. 最早阶段

系统最初是：

- 长视频 + SRT
- 通过 AI 从字幕中筛选内容
- 生成脚本
- 匹配时间轴
- 用 FFmpeg 剪出一条视频

这是典型的单视频 CLI 工作流。

#### 2. skill.py 阶段

随后项目扩成给外部智能体调用的 `skill.py` 流程。

这个阶段重点是：

- 查询视频
- 下载素材
- 生成脚本
- 匹配时间轴
- 生成视频并上传

说明当时项目重心偏向“服务外部智能体调用”。

#### 3. batch_generator 阶段

从 2026-03-29 开始，项目重心明显转向：

- `batch_generator.py`
- 批量生成
- 自动评分
- 筛选候选

当前真正的工作中心已经不是 `skill.py`，而是 `batch_generator.py`。

#### 4. V2 三阶段增强

当前路线图已经明确：

1. 视觉评分
2. 拼接自然度评分
3. 多视频组合生成

目前前两阶段骨架已完成，第三阶段已有最小骨架，但还没有彻底闭环。

## 六、当前明确的协作分工

### 当前决定

从现在开始，协作方式明确为：

- **Codex 负责写代码**
- **Claude / claude-cli 负责架构评估、结构化整理、路线和风险判断**
- **Opencode 只负责测试、验证、跑检查、反馈问题和修测试期暴露的小问题**

### 原因

这样分工有几个目的：

1. 避免多人同时写核心代码导致风格漂移
2. 把 token 集中在真正重要的代码实现上
3. 把架构和执行分开，避免主线混乱
4. 让测试和验证独立成辅助闭环

### 三机协作约定

当前有一个非常关键的运行现实：

- **Codex** 在一台机器上，负责实现代码
- **Claude / claude-cli** 在另一台机器上，负责架构和高层评估
- **Opencode** 在第三台机器上，且它是更接近真实运行环境的那台机器

也就是说：

- 三者不在同一台机器上
- 通过 Git 协作同步代码和文档
- `Opencode` 更接近真实运行环境，因此适合负责测试和验证

### 当前 Git 交互方式

当前默认通过 Git 交互：

- Codex 写代码并提交
- 推送到远端仓库
- 其他机器拉取远端代码
- Claude / Opencode 根据最新提交继续工作或评估

这意味着：

- 文档必须足够清楚
- 提交必须尽量表达真实进度
- 不应依赖“口头上下文”来维持协作

## 七、当前代码风格偏好

当前代码风格有几个核心原则：

1. 目标优先
2. 主链路优先
3. 过程式优先
4. 少 class
5. 少无意义抽象
6. 少无意义中间层
7. 少低价值 try/except
8. 少写几乎永远运行不到的防御代码
9. 每段代码尽量有明确用途

当前不是追求“漂亮架构”或“高级抽象”，而是：

- 用有限资源把主线功能做成
- 避免大的、本质性的错误
- 小错误可以后续通过测试修

一句话总结：

**代码要直、少、清楚、够用，优先把主链路做通。**

## 八、当前踩过的坑

这是给后续评估非常关键的一部分。

### 坑 1：让 Opencode 写第三阶段骨架时，任务写得太偏“结构”，没把“联通”写成硬标准

结果是：

- 它把文件和模块搭起来了
- 范围控制也还行
- 但主链路没有真正打通

这说明：

- 任务里写“最小骨架”是不够的
- 必须写“最小可联通骨架”

### 坑 2：多视频新结构和旧 `keep_intervals` 结构没有提前对齐

这是当前第三阶段最核心的技术坑。

旧系统稳定地依赖：

- `keep_intervals`

但多视频新模块天然更想使用：

- `segment dict`

如果转换层不清楚，会导致：

- 数据契约打架
- 主链路不通
- 输出层难以收口

### 坑 3：不能把 Opencode 当成真正有“时间感”的执行器

之前给它安排过“连续 1 小时自动执行，并且每 10-15 分钟记录一次进度”的要求。

后来实际反馈证明：

- 它可以连续推进任务
- 但不能可靠感知真实时间刻度
- 不适合做定时式进度记录

经验是：

- 以后给长任务时，要按“阶段节点”记录进度
- 不要按真实时间切片记录进度

### 坑 4：核心文档不能堆太多

之前 `core` 放了过多阶段性或一次性文档。

后来重新清洗后确定：

- `core` 只保留长期核心文档
- 阶段细节、一次性评估、交接记录放 `reference`
- 历史计划放 `archive`

## 九、当前最关键的架构问题

这是希望另一个 AI 重点评估的地方。

### 问题 1：第三阶段最小闭环应该闭到哪里

当前有两种可能：

#### 方案 A：闭到候选和 summary

做到：

- 每个视频形成片段池
- 多视频候选能组合出来
- 有评分
- 有 summary
- 暂不做真正跨视频视频输出

优点：

- 风险低
- 容易收敛
- 容易验证方向是否对

#### 方案 B：直接闭到最终多视频视频输出

做到：

- 让多视频候选最终真的生成跨视频成片

优点：

- 功能完整

缺点：

- 风险高
- 复杂度高
- 更可能拖慢当前阶段主线

当前最需要判断的是：

**第三阶段现在应该先停在 A，还是直接做 B。**

### 问题 2：`keep_intervals` 是否应该继续作为底层稳定接口

当前我的倾向是：

- 继续保留 `keep_intervals` 作为底层稳定产物
- 在第三阶段上面加一个薄的转换层
- 不要彻底绕开旧结构

但这是否是最合理的做法，需要外部架构视角再评估一次。

### 问题 3：多视频最终输出层要不要立刻实现

当前单视频输出函数 `cut_video_main()` 已经稳定。

但多视频最终输出如果要真正成立，理论上可能需要：

- 新的多输入视频拼接出口
- 或一个清楚的阶段性降级策略

这也是当前架构上非常关键的判断点。
