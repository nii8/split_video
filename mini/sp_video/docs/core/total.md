## 最高优先级规则：代码写完后必须先 Push，再允许测试

这是当前协作里的最高优先级执行规则之一。

规则内容：

- 只要代码已经写完，并且下一步要交给别人测试
- 就必须先确认代码已经 push 到远端
- 没有 push，不允许进入“让别人测试”的阶段

原因非常直接：

- 如果代码没 push，上游机器拉不到最新实现
- 那么后续测试就是白测
- 会浪费测试机器时间
- 会浪费 token
- 会浪费用户时间
- 还会让用户回来花额外时间排查“为什么测不到最新代码”

这类问题不是小问题，而是：

- 协作事故
- 资源浪费
- 高优先级流程错误

所以以后必须执行下面这个顺序：

1. 写代码
2. 本地自查
3. 提交 git
4. push 到远端
5. 明确确认远端已有对应提交
6. 再通知别人开始测试

如果第 4 步没有完成，就不能让：

- Opencode 测试
- Claude 评估最新实现
- 任何其他机器继续后续流程

一句话：

**没有 push，就等于没有交付；没有交付，就不允许进入测试阶段。**

## 固定协作模板：给 Opencode 的任务口令

为了节省用户时间、减少来回协调、避免遗漏关键步骤，后续默认采用固定任务口令模板。

### 默认执行顺序

以后给 `Opencode` 派任务时，默认必须包含这几个动作：

1. `git pull origin main` 拉取最新代码
2. 阅读指定任务文档
3. 执行指定测试 / 小修 / 验证任务
4. 输出指定 report 文件
5. `git commit`
6. `git push origin main`

如果没有 `git pull`，它可能在旧代码上工作。
如果没有 `git push`，后续测试和协作仍然会白做。

### 通用模板

以后默认用下面这套模板给 `Opencode` 下任务：

```text
先 `git pull origin main` 拉取最新代码。
阅读：`<文档1>`、`<文档2>`。
你只做：`<测试/小修/验证>`。
允许：`<小 bug / 路径 / 参数>`。
不允许：`<改架构 / 重写模块 / 扩功能>`。
最后输出：`<report 路径>`。
完成后：`git commit` 并 `git push origin main`。
```

### 当前适合用户的最省时间模式

用户后续只做 3 件事：

1. 把 Codex 给出的任务口令发给 `Opencode`
2. 等 `Opencode` 跑完后，把结果截图贴给 Codex
3. 由 Codex 负责分析截图、判断下一步

### 用户不再承担的工作

为了节省用户精力，用户不应继续承担：

- 来回转述大量上下文
- 自己判断测试报告是否可信
- 自己排查是不是没 pull / 没 push
- 自己看代码细节决定下一步

### Codex 的职责

当用户把 `Opencode` 的结果截图贴回来后，Codex 应直接负责：

- 先默认 `git pull origin main` 拉取最新远端代码
- 判断它是否真的拉了最新代码
- 判断它是否跑到了目标步骤
- 判断结论是否可信
- 判断问题属于环境、代码、参数还是流程问题
- 明确告诉用户下一步该谁继续做什么

### 结果截图后的默认动作

以后只要用户贴的是：

- `Opencode` 的结果截图
- `Opencode` 的测试结论
- `Opencode` 的提交信息

默认动作不是先空谈分析，而是：

1. 先拉最新远端代码
2. 再结合截图判断当前真实状态
3. 再决定是继续让 `Opencode` 测，还是由 Codex 接着写

一句话：

**截图一来，先拉代码，再分析。**

### 一句话原则

以后协作要尽量压缩成：

**Codex 写任务和判断，Opencode 执行和测试，用户只转发任务口令并把结果截图贴回。**

## 一、项目当前定位

当前项目 `mini/sp_video` 的本质，不再只是一个“输入视频 + 字幕 -> 输出一条短视频”的 CLI 工具。

它已经演进为：

**一个自动短视频候选生成、评分、筛选、输出的本地批处理系统。**

当前真正的主线是：

- 批量生成候选
- 自动评分
- 筛选高质量结果
- 逐步增强视觉判断、拼接判断、多视频组合能力

也就是说，它正在从：

- 单视频自动剪辑工具

演进为：

- 自动短视频编排与筛选引擎

---

## 二、项目主线

项目主线可以概括为：

`CLI 原型 -> skill.py 智能体接口 -> batch_generator 批量预生成系统 -> V2 三阶段增强 -> 当前进入多视频骨架阶段`

### 关键阶段

#### 1. 最早阶段

系统最初是：

- 长视频 + SRT
- 通过 AI 从字幕中筛选内容
- 生成脚本
- 匹配时间轴
- 用 FFmpeg 剪出一条视频

这是典型的单视频 CLI 工作流。

#### 2. skill.py 阶段

随后项目扩成给外部智能体调用的 `skill.py` 流程。

这个阶段重点是：

- 查询视频
- 下载素材
- 生成脚本
- 匹配时间轴
- 生成视频并上传

说明当时项目重心偏向“服务外部智能体调用”。

#### 3. batch_generator 阶段

从 2026-03-29 开始，项目重心明显转向：

- `batch_generator.py`
- 批量生成
- 自动评分
- 筛选候选

当前真正的工作中心已经不是 `skill.py`，而是 `batch_generator.py`。

#### 4. V2 三阶段增强

当前路线图已经明确：

1. 视觉评分
2. 拼接自然度评分
3. 多视频组合生成

目前前两阶段骨架已完成，第三阶段已有最小骨架，但还没有彻底闭环。

## 六、当前明确的协作分工

### 当前决定

从现在开始，协作方式明确为：

- **Codex 负责写代码**
- **Claude / claude-cli 负责架构评估、结构化整理、路线和风险判断**
- **Opencode 只负责测试、验证、跑检查、反馈问题和修测试期暴露的小问题**

### 原因

这样分工有几个目的：

1. 避免多人同时写核心代码导致风格漂移
2. 把 token 集中在真正重要的代码实现上
3. 把架构和执行分开，避免主线混乱
4. 让测试和验证独立成辅助闭环

### 三机协作约定

当前有一个非常关键的运行现实：

- **Codex** 在一台机器上，负责实现代码
- **Claude / claude-cli** 在另一台机器上，负责架构和高层评估
- **Opencode** 在第三台机器上，且它是更接近真实运行环境的那台机器

也就是说：

- 三者不在同一台机器上
- 通过 Git 协作同步代码和文档
- `Opencode` 更接近真实运行环境，因此适合负责测试和验证

### 当前 Git 交互方式

当前默认通过 Git 交互：

- Codex 写代码并提交
- 推送到远端仓库
- 其他机器拉取远端代码
- Claude / Opencode 根据最新提交继续工作或评估

这意味着：

- 文档必须足够清楚
- 提交必须尽量表达真实进度
- 不应依赖“口头上下文”来维持协作

## 七、当前代码风格偏好

当前代码风格有几个核心原则：

1. 目标优先
2. 主链路优先
3. 过程式优先
4. 少 class
5. 少无意义抽象
6. 少无意义中间层
7. 少低价值 try/except
8. 少写几乎永远运行不到的防御代码
9. 每段代码尽量有明确用途

当前不是追求“漂亮架构”或“高级抽象”，而是：

- 用有限资源把主线功能做成
- 避免大的、本质性的错误
- 小错误可以后续通过测试修

一句话总结：

**代码要直、少、清楚、够用，优先把主链路做通。**

## 八、当前踩过的坑

这是给后续评估非常关键的一部分。

### 坑 1：让 Opencode 写第三阶段骨架时，任务写得太偏“结构”，没把“联通”写成硬标准

结果是：

- 它把文件和模块搭起来了
- 范围控制也还行
- 但主链路没有真正打通

这说明：

- 任务里写“最小骨架”是不够的
- 必须写“最小可联通骨架”

### 坑 2：多视频新结构和旧 `keep_intervals` 结构没有提前对齐

这是当前第三阶段最核心的技术坑。

旧系统稳定地依赖：

- `keep_intervals`

但多视频新模块天然更想使用：

- `segment dict`

如果转换层不清楚，会导致：

- 数据契约打架
- 主链路不通
- 输出层难以收口

### 坑 3：不能把 Opencode 当成真正有“时间感”的执行器

之前给它安排过“连续 1 小时自动执行，并且每 10-15 分钟记录一次进度”的要求。

后来实际反馈证明：

- 它可以连续推进任务
- 但不能可靠感知真实时间刻度
- 不适合做定时式进度记录

经验是：

- 以后给长任务时，要按“阶段节点”记录进度
- 不要按真实时间切片记录进度

### 坑 4：核心文档不能堆太多

之前 `core` 放了过多阶段性或一次性文档。

后来重新清洗后确定：

- `core` 只保留长期核心文档
- 阶段细节、一次性评估、交接记录放 `reference`
- 历史计划放 `archive`

## 九、当前最关键的架构问题

这是希望另一个 AI 重点评估的地方。

### 问题 1：第三阶段最小闭环应该闭到哪里

当前有两种可能：

#### 方案 A：闭到候选和 summary

做到：

- 每个视频形成片段池
- 多视频候选能组合出来
- 有评分
- 有 summary
- 暂不做真正跨视频视频输出

优点：

- 风险低
- 容易收敛
- 容易验证方向是否对

#### 方案 B：直接闭到最终多视频视频输出

做到：

- 让多视频候选最终真的生成跨视频成片

优点：

- 功能完整

缺点：

- 风险高
- 复杂度高
- 更可能拖慢当前阶段主线

当前最需要判断的是：

**第三阶段现在应该先停在 A，还是直接做 B。**

### 问题 2：`keep_intervals` 是否应该继续作为底层稳定接口

当前我的倾向是：

- 继续保留 `keep_intervals` 作为底层稳定产物
- 在第三阶段上面加一个薄的转换层
- 不要彻底绕开旧结构

但这是否是最合理的做法，需要外部架构视角再评估一次。

### 问题 3：多视频最终输出层要不要立刻实现

当前单视频输出函数 `cut_video_main()` 已经稳定。

但多视频最终输出如果要真正成立，理论上可能需要：

- 新的多输入视频拼接出口
- 或一个清楚的阶段性降级策略

这也是当前架构上非常关键的判断点。

## 十九、自动推进规则

为了尽量解放用户时间，后续默认按“自动推进”协作，不再频繁等待用户逐步确认。

### 用户负责什么

用户只负责：

1. 给最终目标和优先级
2. 在方向发生变化时拍板
3. 看阶段结果和关键风险

用户不再负责：

- 手动提醒先 pull / 先 push
- 手动协调 Codex 和 Opencode
- 手动判断下一步该谁做
- 反复检查有没有同步远端代码

### Codex 默认自动做什么

在没有新的方向性冲突时，Codex 默认自动执行下面流程：

1. 先同步远端最新代码
2. 读取最新任务文档 / 测试报告
3. 判断当前主矛盾
4. 决定下一步是：
   - 自己继续写代码
   - 还是给 Opencode 下测试 / 小修任务
5. 如果是自己写代码：
   - 直接实现
   - 本地做最小自查
   - 提交 git
   - push 到远端
6. 如果是给 Opencode 任务：
   - 先写清任务文档
   - 提供可直接转发的短口令
   - 等用户贴回结果截图
7. 用户贴回截图后：
   - 默认先拉最新远端代码
   - 再分析结果
   - 再继续推进

### 只有这些情况才停下来问用户

只有遇到下面这些情况，Codex 才应该停下来问用户：

1. 需要改变整体方向
2. 需要改变当前阶段边界
3. 需要做明显超预算的大改
4. 发现当前方案本身走不通
5. 需要删除或放弃一条重要路线

除此之外，默认继续往前推进。

### 当前推荐的自动推进模式

当前项目最适合的自动推进模式是：

- Codex：主实现 + 主判断 + Git 提交和 push
- Opencode：运行环境测试 + 小修 + 测试报告
- Claude / claude-cli：阶段性架构评估
- 用户：只看结果、拍板方向、贴回截图

### 一句话原则

**用户给目标，Codex 自动推进主线，Opencode 负责环境验证，非方向性问题默认不停顿。**

## 二十一、当前已对齐的协作流程

这部分是当前最重要的默认协作流程，后续应长期复用。

### Step 1：Codex 自动推进代码

Codex 默认执行：

1. 先拉远端最新代码
2. 判断当前主矛盾
3. 围绕目标直接写代码
4. 本地做最小自查
5. 提交 git
6. push 到远端

### Step 2：Codex 生成测试任务

Codex 完成当前一轮代码后：

1. 写清测试任务文件
2. 写出可直接转发给 `Opencode` 的短提示词

### Step 3：用户最小参与

用户只做：

1. 把任务提示词发给 `Opencode`

不需要做：

- 自己整理上下文
- 自己解释代码意图
- 自己判断测试边界

### Step 4：Opencode 执行测试

`Opencode` 默认执行：

1. `git pull origin main`
2. 阅读指定文档
3. 执行测试 / 验证 / 小修
4. 输出测试报告
5. `git commit`
6. `git push origin main`

### Step 5：用户贴截图

用户拿到 `Opencode` 的结果后：

1. 截图
2. 发给 Codex

### Step 6：Codex 自动接续

当用户贴回截图后，Codex 默认执行：

1. 先拉最新远端代码
2. 自动分析截图和远端状态
3. 自动判断下一步：
   - 继续写代码
   - 修 bug
   - 或继续给 `Opencode` 新测试任务

### 一句话版本

**Codex 写代码并推进，Opencode 测试并反馈，用户只转发任务和贴回截图。**

## 二十二、Core 自动维护规则

从现在开始，`docs/core/` 不是静态文档区，而是由 Codex 自动维护的长期核心知识区。

### Codex 默认职责

Codex 应自动负责：

1. 把关键对齐结果写入 `core`
2. 把新的高价值协作经验写入 `core`
3. 发现 `core` 里有过时内容时主动清理
4. 发现重复内容时主动压缩
5. 随项目推进更新核心价值观、规则和流程

### Core 的维护原则

`core` 只保留：

- 长期有效
- 高复用
- 高价值
- 会反复指导后续协作和实现的内容

不应长期保留：

- 一次性说明
- 低价值阶段记录
- 已经过时的旧规则
- 和当前主线关系很弱的内容

### Core 价值观不是固定不变的

核心价值观要随着项目推进不断：

- 成长
- 升级
- 进化

也就是说：

- 旧规则如果已经不适合当前阶段，要主动替换
- 新经验如果已经证明更有效，要提升到核心层
- 协作流程如果能进一步节省用户时间，要更新到 `core`

### 一句话原则

**Codex 不只是使用 `core`，还要持续维护 `core`，让它始终代表当前最有效的目标、规则和协作方式。**
